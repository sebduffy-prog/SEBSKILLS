# /sebduffy — find the right skill, then do the work

Type `/sebduffy <what you want>` → route to the best skill in the library and **just do it**.

## How to operate (perform silently — the user sees none of these steps)

1. **Match** the request to the single best skill in the **Catalogue** below, by name / keywords / intent.
2. **Load** its instructions with the first that works:
   **bundled `library/<name>/SKILL.md`** (a file inside this skill's own folder — present when installed as
   the self-contained plugin; needs no network, so it works in sandboxes and offline) →
   local `~/.claude/skills/<name>/SKILL.md` → local `./.claude/skills/<name>/SKILL.md` →
   fetch `https://raw.githubusercontent.com/sebduffy-prog/SEBSKILLS/main/skills/<category>/<name>/SKILL.md`
   (WebFetch, or `curl -fsSL`). If the skill isn't in the Catalogue (newer than this file) or you need its
   category for the URL, read the bundled `library/manifest.json` first, else fetch
   `https://raw.githubusercontent.com/sebduffy-prog/SEBSKILLS/main/manifest.json`, and look it up there.
   Never invent a skill's contents; if you truly can't load it, say so and print the URL.
3. **Do the task**, following that skill. If it references bundled files (`scripts/…`, `assets/…`,
   `references/…`), load each by its exact relative path the same way — bundled `library/<name>/<relative-path>`,
   else local, else `…/skills/<category>/<name>/<relative-path>` (there is no remote directory listing, so only
   fetch paths the skill body names).

## Output discipline (this is the whole point)

**Action-first and quiet. Deliver the result, not the routing.**
No catalogue dump, no match scores, no fetch-ladder talk, no "I'll now use…", no summarising the skill before doing it.
Preamble = **at most one short line** (`→ quick-dashboard`) or nothing. Then act.
Ask a question **only** when the request is genuinely ambiguous or too thin to act on.

This discipline binds the **routed skill's execution too**, not just the routing: the deliverable is the
**real artifact** the skill promises (a file written, a command run, a render produced) — verified to exist
before you report it — never a description, plan, or walkthrough of what the artifact *would* be. If inputs
are missing, ship the skill's awaiting-data scaffold rather than prose. Text is the deliverable only for
skills whose `produces:` genuinely is prose (a strategy, a review) — and then it's the *finished* prose.

## Commands & fallbacks (keep these terse too)

- `/sebduffy list` or `/sebduffy <category>` → show that slice of the Catalogue (this is the only time you print it).
- `/sebduffy <exact-skill-name>` → load it directly, skip matching.
- **Two skills tie** → name both in one line, ask which. **Too vague** → load `requirement-elicitation`, ask 1–3 sharp questions, then route. **No fit** → load `skill-gap-detector` (offer `skill-creator`). **Multi-part job** → load `skill-chaining-composer` to sequence skills.
